# API Contract — v1 (`/api/v1/...`)

Created: 2026-04-07  
Status: Draft contract (implementation target)

This document defines the HTTP API contract for v1 of the Z-Wave QR vault service.
It is intentionally explicit so dashboard/settings UIs and automation clients can be built against it before full backend completion.

## 1) Versioning and backward compatibility

- All stable v1 endpoints are namespaced under `/api/v1/...`.
- `v1` is **backward-compatible for additive changes**:
  - Allowed without breaking change:
    - Adding new optional request fields.
    - Adding new optional response fields.
    - Adding new enum values where documented as extensible.
  - Not allowed in v1 minor revisions:
    - Removing fields.
    - Renaming fields.
    - Changing field types.
    - Changing required/optional semantics of existing fields.
    - Changing default pagination/sort behavior.
- Breaking changes require a new version namespace (e.g., `/api/v2/...`).
- Every stored record includes `schema_version` (`"1"`) for payload-level evolution.

## 2) Common conventions

### Content type

- Request and response body format: `application/json`, except QR render endpoint (`image/png`).

### Time format

- All timestamps are RFC 3339 / ISO-8601 UTC (`YYYY-MM-DDTHH:MM:SSZ`).

### Error format

Validation and domain errors use a normalized shape:

```json
{
  "error": {
    "code": "validation_error",
    "message": "Request validation failed",
    "details": [
      {
        "field": "dsk",
        "reason": "must be unique; value already exists",
        "type": "value_error.conflict"
      }
    ],
    "request_id": "req_01H...",
    "timestamp": "2026-04-07T12:34:56Z"
  }
}
```

Error code guidance:

- `validation_error` → `400` (or `422` if framework-native validation path is preserved)
- `not_found` → `404`
- `conflict` → `409`
- `sync_in_progress` → `409`
- `git_failure` → `502`
- `internal_error` → `500`
- `unauthorized` → `401`
- `forbidden` → `403`

`request_id` SHOULD be echoed from `X-Request-ID` if provided, else server-generated.

### 4.2 CRUD — Devices

#### `GET /api/v1/devices`

List devices with search/filter/sort/pagination.

**Query params**

- `q` (string, optional): case-insensitive substring search over `device_name`, `dsk`, and `description`/notes.
- `name` (string, optional): case-insensitive substring match for `device_name`.
- `dsk` (string, optional): normalized partial match (hyphens ignored for matching).
- `notes` (string, optional): case-insensitive substring over description/notes.
- `sort` (string, optional): one of:
  - `updated_at`
  - `created_at`
  - `device_name`
  - `dsk`
  - `sync_state`
- `order` (string, optional): `asc` or `desc`.
- `page` (integer, optional, min=1), default `1`.
- `per_page` (integer, optional, min=1, max=200), default `25`.

**Default list semantics**

- Sort default: `updated_at`
- Order default: `desc`
- Pagination default: `page=1&per_page=25`

**200 Response**

```json
{
  "items": [
    {
      "schema_version": "1",
      "id": "dev-4f93ab9c12d0e9bb5a31",
      "device_name": "Living Room Dimmer",
      "raw_value": "900132782003...",
      "dsk": "12345-67890-12345-67890-12345-67890-12345-67890",
      "description": "South wall",
      "created_at": "2026-04-07T10:00:00Z",
      "updated_at": "2026-04-07T10:05:00Z",
      "sync": {
        "state": "synced",
        "last_success_at": "2026-04-07T10:05:13Z",
        "last_error": null,
        "head_commit": "a1b2c3d"
      }
    }
  ],
  "pagination": {
    "page": 1,
    "per_page": 25,
    "total_items": 42,
    "total_pages": 2
  },
  "sort": {
    "field": "updated_at",
    "order": "desc"
  },
  "applied_filters": {
    "q": "dimmer",
    "name": null,
    "dsk": null,
    "notes": null
  }
}
```

#### `POST /api/v1/devices`

Create a device record.

**Request body**

```json
{
  "raw_value": "900132782003...",
  "device_name": "Living Room Dimmer",
  "location": "Living Room",
  "description": "South wall",
  "manufacturer": "Acme",
  "model": "ZW-1000",
  "metadata": {
    "favorite": true
  }
}
```

Server derives and validates: `id`, `dsk`, `schema_version`, timestamps.

**201 Response**

- Returns full Device resource.
- Header: `Location: /api/v1/devices/{id}`

**Errors**

- `400/422 validation_error` (invalid raw payload)
- `409 conflict` (duplicate `raw_value` or `dsk`)

---

#### `GET /api/v1/devices/{id}`

Fetch one record by ID.

**200 Response**

- Full Device resource.

**404 Response**

- Standard error format with `code=not_found`.

---

#### `PUT /api/v1/devices/{id}`

Replace mutable metadata fields.

Mutable fields in v1:

- `device_name`
- `location`
- `description`
- `manufacturer`
- `model`
- `metadata`

Identity fields are immutable in this route (`id`, `raw_value`, `dsk`, `schema_version`).

**Request body (example)**

```json
{
  "device_name": "Living Room Main Dimmer",
  "description": "Updated label",
  "metadata": {
    "favorite": true,
    "room_id": "lr"
  }
}
```

**200 Response**

- Full updated Device resource.

**Errors**

- `400/422 validation_error`
- `404 not_found`

---

#### `DELETE /api/v1/devices/{id}`

Delete one record.

**204 Response**

- Empty body.

**Errors**

- `404 not_found`

### 4.3 Import

#### `POST /api/v1/import`

Bulk import records from array input.

**Request body**

```json
{
  "items": [
    {
      "raw_value": "900132782003...",
      "device_name": "Kitchen Sensor",
      "description": "imported"
    },
    {
      "raw_value": "900132782004...",
      "device_name": "Garage Door Sensor"
    }
  ],
  "mode": "upsert",

- `create_only` (fail duplicates)
- `upsert` (update mutable fields on conflicts)

**200 Response**

    "created": 1,
    "updated": 1,
    "skipped": 0,
    "failed": 0

      "id": "dev-aaa..."
    },
    {
      "index": 1,
      "status": "updated",
      "id": "dev-bbb..."
    }
  ]
}
```

### 4.4 QR render

#### `GET /api/v1/devices/{id}/qr.png`

Render a QR image from canonical stored payload.

**Query params**

**200 Response**

- `Content-Type: image/png`
- Binary PNG body

**Errors**

- `404 not_found`

### 4.5 Sync trigger and status

#### `POST /api/v1/sync`

Manually triggers pull/rebase + pending commit/push cycle.

**Request body (optional)**

```json
{
  "reason": "user_triggered",
  "force": false
}
```

**202 Response**

```json
{
  "job_id": "sync_01H...",
  "accepted": true,
  "state": "queued",
  "requested_at": "2026-04-07T10:10:00Z"
}
```

If synchronous mode is configured, response MAY be `200` with completed status payload.

## 5) Sync metadata fields required by UI

The following fields are required contracts because dashboard and settings pages depend on them.

### Dashboard-required fields

At list/detail level (`Device.sync`):

- `state` (`synced|pending_push|syncing|failed|conflicted`)
- `last_attempt_at` (nullable timestamp)
- `last_success_at` (nullable timestamp)
- `last_error` (nullable string)
- `head_commit` (short SHA, nullable)
- `branch` (string)

### Settings-required fields

From `GET /api/v1/sync/status`:

- `mode`
- `repo.url_masked`
- `repo.branch`
- `repo.head_commit`
- `repo.ahead`
- `repo.behind`
- `repo.dirty`
- `timing.interval_seconds`
- `timing.last_attempt_at`
- `timing.last_success_at`
- `timing.next_scheduled_at`
- `timing.last_duration_ms`
- `last_error`
- `counters.success_count`
- `counters.failure_count`

Fields above are part of the v1 compatibility guarantee.

## 6) Search/filter semantics details

Normalization rules (v1):

- Text search (`q`, `name`, `notes`) is case-insensitive, Unicode-aware substring match after trimming.
- DSK search ignores hyphen separators in both stored and query values (e.g., `12345-678` equals `12345678`).
- Multiple specific filters (`name`, `dsk`, `notes`) combine with logical `AND`.
- `q` is an additional broad filter across `device_name`, `dsk`, and `description`; if combined with specific filters, it is also `AND`ed.

Deterministic sort tiebreakers:

1. Requested sort field
2. `id` ascending as final tie-breaker

This ensures stable pagination across requests.

  "device_name": "Hallway Sensor",
